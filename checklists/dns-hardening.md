# DNS Hardening Checklist

Verification steps for DNS-layer protections that affect web traffic. Use after initial setup and quarterly thereafter.

## Pre-flight

- [ ] Domain registrar account is accessible
- [ ] DNS host account is accessible (often the same as registrar, but not always)
- [ ] You know which CA issues your TLS certificates (Let's Encrypt, Sectigo, DigiCert, etc.)
- [ ] You know which mail provider (if any) sends mail for the domain

## DNSSEC

DNSSEC requires coordination between the DNS host and the registrar.

### Enablement

- [ ] DNSSEC is enabled at the DNS host (Cloudflare, Route 53, BIND, etc.)
- [ ] DNS host has generated a DS (Delegation Signer) record
- [ ] DS record is published at the domain registrar
- [ ] Registrar confirms DNSSEC is "active" (this may take 24 hours to propagate)

### Validation

- [ ] [dnssec-analyzer.verisignlabs.com](https://dnssec-analyzer.verisignlabs.com/) shows green checks at every level
- [ ] [dnsviz.net](https://dnsviz.net/) shows a chain of trust from root to the domain
- [ ] `dig +dnssec example.com` shows RRSIG records in the response
- [ ] `dig +dnssec +cd example.com` (cd = checking disabled) succeeds — confirms records resolve when validation is bypassed

### Recovery plan

- [ ] Recovery procedure for broken DNSSEC is documented
- [ ] Recovery procedure assumes registrar access is available
- [ ] Procedure is: remove DS record at registrar, wait for caches to expire (TTL of DS), then re-enable

## CAA — Certificate Authority Authorization

### Initial deployment

- [ ] Identified all CAs that have issued certificates for the domain in the last year (check [crt.sh](https://crt.sh) by domain)
- [ ] List of authorized CAs is finalized
- [ ] CAA record is added at the root domain (applies to subdomains by default)
- [ ] `issue` records list all authorized CAs
- [ ] `issuewild` records list CAs allowed to issue wildcards (often a subset)
- [ ] `iodef` record points to a monitored email address

### Validation

- [ ] `dig CAA example.com` returns the expected records
- [ ] [SSL Labs SSL Test](https://www.ssllabs.com/ssltest/) "CAA Record" section shows the records correctly
- [ ] Certificate renewal succeeds AFTER the CAA record is in place (the most important test)

### Common mistakes

- [ ] CAA record was not set in a way that EXCLUDES your actual CA — verify the issuer identifier matches:
  - Let's Encrypt: `letsencrypt.org`
  - Sectigo: `sectigo.com`
  - DigiCert: `digicert.com`
  - Amazon: `amazon.com`
  - Google Trust Services: `pki.goog`
- [ ] If wildcard certificates are used, `issuewild` record is also set

## SPF — Sender Policy Framework

### Initial deployment

- [ ] Decision is made: does this domain send mail?
- [ ] If NO: SPF record is `v=spf1 -all` (hard reject all mail claiming to be from this domain)
- [ ] If YES: SPF record includes ALL legitimate sending sources
- [ ] Common includes are added where applicable:
  - Google Workspace: `include:_spf.google.com`
  - Microsoft 365: `include:spf.protection.outlook.com`
  - Mailgun: `include:mailgun.org`
  - SendGrid: `include:sendgrid.net`
  - Amazon SES: `include:amazonses.com`
- [ ] SPF record uses `~all` (soft fail) initially during testing
- [ ] SPF record is tightened to `-all` (hard fail) only after testing confirms it is complete

### Validation

- [ ] `dig TXT example.com | grep spf1` returns the expected record
- [ ] [MXToolbox SPF check](https://mxtoolbox.com/spf.aspx) shows no errors or warnings
- [ ] DNS lookup count stays under the 10-lookup limit (each `include:` counts)
- [ ] Test mail from each sending source delivers to Gmail without going to spam

## DKIM — DomainKeys Identified Mail

### Initial deployment

DKIM is generated by the mail provider, not manually.

- [ ] Each mail provider's DKIM record is added to DNS
- [ ] Selector subdomain matches what the mail provider expects (`google._domainkey`, `mailgun._domainkey`, etc.)
- [ ] DKIM record is published at the correct selector

### Validation

- [ ] `dig TXT <selector>._domainkey.example.com` returns the public key
- [ ] Mail provider's DKIM verification page shows the record as valid
- [ ] Test mail from the provider's "Show original" or "Headers" view shows `DKIM=pass`

## DMARC — Domain-based Message Authentication

### Initial deployment

- [ ] Aggregate reporting address is set up (`rua=mailto:dmarc-reports@example.com`)
- [ ] Forensic reporting address is set up if desired (`ruf=mailto:dmarc-forensic@example.com`)
- [ ] Policy is initially set to `p=none` (monitoring only)
- [ ] DMARC record is published at `_dmarc.example.com`

### Validation

- [ ] `dig TXT _dmarc.example.com` returns the expected record
- [ ] [MXToolbox DMARC check](https://mxtoolbox.com/dmarc.aspx) shows the record as valid
- [ ] Aggregate reports start arriving (typically within 1-2 days)

### Progression

After 30-60 days of monitoring:

- [ ] DMARC reports have been reviewed
- [ ] All legitimate senders pass SPF or DKIM (or both)
- [ ] Identified failing legitimate senders have been fixed (added to SPF, DKIM configured)
- [ ] Policy progresses to `p=quarantine` with `pct=10` (10% of failing mail quarantined)
- [ ] Monitor for 30 days
- [ ] Increase `pct` to 50, then 100
- [ ] Monitor for 30 days
- [ ] Policy progresses to `p=reject`

### Common mistakes

- [ ] DMARC `p=reject` enabled before SPF and DKIM are complete — blocks legitimate mail
- [ ] DMARC alignment mode (`adkim=`, `aspf=`) set to strict before testing — narrows what passes alignment
- [ ] No reporting address configured — no visibility into failures

## Subdomain Policy

DMARC `p=` applies to the organizational domain. The `sp=` (subdomain policy) parameter governs subdomains independently.

- [ ] `sp=reject` is considered for subdomains that should never send mail
- [ ] Subdomains that legitimately send mail have their own SPF and DKIM records

## Recurring Review

Schedule quarterly:

- [ ] CAA records still match the current set of authorized CAs (check past 12 months of certs at [crt.sh](https://crt.sh))
- [ ] SPF record still covers all current sending sources
- [ ] DKIM keys have been rotated within the last 12 months (mail providers often handle this automatically)
- [ ] DMARC reports continue to be reviewed
- [ ] DNSSEC validation still passes

## Documentation

- [ ] All DNS records in this checklist are documented in a runbook
- [ ] DNSSEC enablement process and recovery process are documented
- [ ] Mail provider DKIM rotation schedule is known
- [ ] Next quarterly review date is recorded
